# Building a Windows Server Infrastructure from Scratch

For the **Server Administration (SAD400)** course at University West, my teammate David and I were tasked with designing and deploying a complete Windows Server infrastructure for a fictional company called *Lejon Family Company*. The project ran over roughly five weeks and covered everything from domain controllers and DNS to GPOs, backup routines, and WSUS — all virtualized on VMware vSphere.

## The Brief

The assignment simulated a real enterprise scenario: build a production-ready Windows environment across two physically separated sites (simulated via VMware), with full redundancy, security policies, file sharing, remote access, update management, a web server, and documented operational routines for every common admin task. The whole system had to be recoverable within 30 minutes if a server went down, and the documentation had to be thorough enough that another administrator could maintain it.

All infrastructure ran on the University West VMware vSphere environment (Cluster-H), with our group allocated the `193.10.237.16/28` address range on VLAN 37.

Both domain controllers sit on the same `/24` subnet but are logically separated into two Active Directory sites. DC1 runs **Server Core** — one of the requirements was that at least one server strip away all unnecessary payload — which gives it a smaller attack surface and lower resource footprint. DC2 runs Desktop Experience since it hosts IIS, WSUS, home folders, and serves as the primary management interface. VMware Tools is installed on every VM for performance and guest management.

## Domain & Active Directory Design

The domain `group1.lab.hv.se` mirrors a company with four departments across two sites:

**Site 1 Management:** Djikstra (Economy boss), Eva (IT boss), Frida (Production boss), and Greta (CEO — boss over all departments).

**Site 2 Management:** Assar (Economy boss), Beta (IT boss), and Ceasar (Production boss).

Each of the three operational departments — **Economy**, **Production**, and **IT** — has five employees who work alternating between both sites. In total we created 22 user accounts across these OUs, each following a naming convention based on first-initial + last-name abbreviation (e.g., `eale` for Earl Lejon).

A key design choice was **separating admin tiers**. Only two IT users (Ida and Isak) hold Domain Admin privileges and can manage Active Directory and Hyper-V on the DCs. The remaining IT staff get local admin rights on workstations through a Restricted Groups GPO (`AssignedLocalAdmins`) but can't touch the domain controllers directly. Everyone outside IT falls into a "Non Admins" group with restricted install permissions.

---

## Remote Access & Virtual Workstations

Three virtual Windows 10 workstations were set up as shared remote desktops. All employees **except Production department members** can connect to them remotely via RDP. This was enforced through a `RemoteAccess` security group — Production users are simply excluded from it.

When users connect, they land in their own environment with access to a **private home folder** (100 MB quota per user) stored centrally on DC2. Permissions are locked so users can only see their own directory — no browsing other people's files. The home folders map automatically on logon.

---

## File Sharing, DFS & FTP

**Home Folders** — Each user gets a 100 MB home folder on DC2 at `\\dc2\homes$\%username%`, accessible only by the owning user. Permissions are set to Read & Execute plus List Folder Contents on just their own folder.

**Shared Folder** — A shared folder (`\\group1.lab.hv.se\delad`, mapped as the M: drive) is accessible by all employees with Full Control. This is replicated across both DCs using **DFS** for redundancy. A logon script (`delad_netlogon.cmd`) and a GPO drive map both handle the automatic mapping.

**FTP Access** — The shared folder is also accessible via FTP on port 21 (hosted on DC2 via IIS). Employees authenticate with their domain credentials using Basic Authentication. The FTP root points to `C:\Delad`, the same volume as the shared folder.

## IIS Web Server

DC2 hosts IIS 10 with a few sites:

The **Intranet Website** on port 8888 displays our group name and hosts the latest version of the system documentation as a downloadable PDF. This satisfies the requirement that the documentation is always accessible to the organization.

**FTP Group1** on port 21 provides the FTP server tied to the shared folder. The **WSUS Administration** site on ports 8530/8531 serves as the management interface for Windows Server Update Services.

All sites are restricted to the local network.

## GPOs — The Backbone of Policy Enforcement

Group Policy was where a lot of the real work happened. Here's a breakdown of every GPO we deployed:

**Password & Lockout Policy (`PasswordPolicy`)** — 10-character minimum length, complexity enabled (must include numbers), 100-day max age, 60-day minimum age, 5 passwords remembered in history. Account lockout kicks in after 5 failed attempts with a 5-minute lockout duration and reset counter. Administrators can manually unlock accounts if needed.

**Software Distribution (`GlobalComputerSoftware`)** — GPO-based software installation pushes **Firefox**, **Adobe Reader**, and **Davisoft Calculator** to all workstations as assigned packages. These install automatically on reboot — satisfying the requirement for an additional browser, a PDF reader, and a program of our choice.

**Economy-Specific Software (`EconomyDepartmentSoftware`)** — **7-Zip** is assigned exclusively to Economy department users through a separate GPO scoped to the Economy OU and workstation group. No other departments see this software, fulfilling the "choose for themselves" program requirement.

**Windows Installer Lockdown (`DissableInstall`)** — Non-admin users can't run Windows Installer, preventing unauthorized software installs. Enforced through a GPO filtered to the `Non Admins` security group.

**Local Admin Assignment (`AssignedLocalAdmins`)** — Uses Restricted Groups to add the `IT_local_admins` group as local administrators on all workstations. This gives all IT staff local admin rights on client machines while keeping domain controller access restricted to only the two Domain Admin accounts.

**IT Server Access (`IT_Server_Access`)** — On the domain controllers, MMC is locked to author mode and only a specific whitelist of snap-ins is allowed: AD Users & Computers, AD Sites & Services, and AD Domains & Trusts. An extra registry setting enables Hyper-V management. This is scoped to the `IT Domain Admins` group (Ida and Isak only).

**Drive Mapping (`Map Network Device`)** — Automatically maps the shared folder as the M: drive for all authenticated users via both a logon script and GPO preference drive maps.

**WSUS Client Targeting (`Updates_for_clients` / `Updates_For_servers`)** — Separate GPOs for workstations and DC1 point machines to the local WSUS instance on DC2. Update detection runs every 22 hours, with notify-before-install behavior so admins can review before applying.

## Windows Update Services (WSUS)

All patching is centralized through WSUS running on DC2. The server synchronizes with Microsoft Update twice daily (first sync at 12:14) and covers products including Active Directory, Microsoft Advanced Threat Analytics, Microsoft Security Essentials, and a selection of Windows update categories.

Updates must be **approved** before they download to clients — we didn't set up auto-approval to keep control over what gets deployed. The WSUS clients include DC1 (Server 2022 Standard) and all three workstations (Windows 10 Pro). DC2 itself updates directly since it hosts the WSUS role.

## DNS Configuration

DNS runs on both DCs with **zone replication** enabled. The domain zone is `group1.lab.hv.se`. Every machine has both forward (A) and reverse (PTR) records:

| Host | IP |
|---|---|
| dc1.group1.lab.hv.se | 193.10.237.16 |
| dc2.group1.lab.hv.se | 193.10.237.19 |
| workstation1.group1.lab.hv.se | 193.10.237.17 |
| workstation2.group1.lab.hv.se | 193.10.237.18 |
| workstation3.group1.lab.hv.se | 193.10.237.20 |

**Recursive queries are disabled** except for the local network — this was a hard requirement to prevent the DNS server from being used in amplification attacks or leaking internal resolution data.

## Backup & Disaster Recovery

The spec required daily offsite backups and a **30-minute recovery target** if a server goes down.

**Daily Backups** — Both DCs run Windows Server Backup on a daily schedule at 23:00, performing full-server backups to a network share on the *other* DC. So DC1's backup lives on DC2 and vice versa — simple offsite redundancy within the lab constraints.

**vSphere Templates** — We created VM templates from both DCs in a known-good state (January 2025 baseline). If a DC fails completely, the recovery process is: deploy the template from vCenter (right-click → New VM from Template → select Cluster-H and SAN-disk production), boot it up, then restore the latest backup on top. This gets us back to operational within the 30-minute window.

**Caveat:** Network-folder backups in Windows Server Backup overwrite the previous backup, so you only ever have the most recent one. In a real production environment I'd want versioned backups or a dedicated backup solution. We also noted that if a backup is corrupted, restoring it onto a template could propagate the issue — so template-only recovery is the fallback.

## Operational Routines

A major part of the deliverable was writing step-by-step runbooks that another admin could follow without guessing:

**Creating a New User** — Copy the template user in ADUC, fill in name and logon credentials, set a temporary password with "must change at next logon," move the account to the correct department OU, and add them to the `ALLA` group (plus `RemoteAccess` if they're not in Production).

**Setting Up a New Virtual Workstation** — Create the VM in vSphere, install Windows, configure DNS to point at both DCs, enable Remote Desktop, join the domain (`group1.lab.hv.se`), then move the computer object from the default Computers container to the Workstations OU so GPOs apply. The A record gets created automatically on domain join.

**Publishing Software via GPO** — Download the `.msi` to the software distribution share on DC2, edit the appropriate GPO (global or department-specific), add the package under Computer or User Configuration → Software Installation, and choose Assigned (auto-install) or Published (user-choice). Reboot workstations to apply.

**Managing Employee Offboarding** — Disable the account in ADUC first (don't delete immediately). Change the password and log in as the user to check for important files in the home folder. Wait at least one week, then delete the account and remove the home folder from `H:\Home Folder\`.

**Running Manual Backups** — Open `wbadmin`, select Backup Once, choose Full Server, target the remote shared folder on the other DC, authenticate with admin credentials, and verify success in the status pane.

**Recreating a Domain Controller** — Deploy from the vSphere template, which comes pre-configured with IP and services from the January 2025 baseline, then restore the latest daily backup. Recommended to update templates after any significant infrastructure changes.

## Reflections

This project was my first real hands-on experience building a multi-server Windows domain from the ground up. A few takeaways:

**Server Core is worth the learning curve.** Managing DC1 entirely through PowerShell and remote tools was harder at first, but it reinforced that GUIs are a convenience, not a requirement. In a security-conscious environment, the reduced attack surface matters.

**GPOs are powerful but messy.** Getting the right combination of security filtering, OU linking, and policy inheritance took a lot of trial and error. The debugging process — `gpresult /r`, `rsop.msc`, checking event logs — became second nature by the end.

**Tiered admin access matters.** Separating "local admin on workstations" from "domain admin on DCs" is a fundamental security principle, and implementing it with Restricted Groups and MMC snap-in restrictions made the concept concrete rather than theoretical.


**The 30-minute recovery target changes how you build.** Knowing that you have to restore a dead server in half an hour forces you to think about templates, backup strategy, and documentation *before* something breaks. It turned disaster recovery from an afterthought into a design constraint.

The full report covers every detail — user lists, share permissions, GPO breakdowns, DNS records, and all the operational routines. This was one of the most comprehensive projects in the program, and it tied together concepts from networking, operating systems, and security courses into a single working system.
